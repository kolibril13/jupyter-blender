JUPYTER-BLENDER — EXPLANATION VIDEO SCRIPT
================================================================================


────────────────────────────────────────────────────────────────────────────────
SECTION 1: MOTIVATION
────────────────────────────────────────────────────────────────────────────────

[SCREEN: Blender scripting editor open, some Python code visible]

If you have ever tried to use Python to automate something in Blender, you know
the workflow is a little awkward. You open the scripting editor inside Blender,
you write your code there, you hit Run Script, and you see the result. That is
fine for small experiments, but it does not feel like a real development
environment. There is no cell-by-cell execution, no inline output, no
autocomplete from a language server. It does not feel like the Python workflow
most of us are used to.

The natural solution would be to open a Jupyter notebook and work from there.
But here is the problem: Blender's Python API lives inside the `bpy` module, and
`bpy` only exists inside the Blender process itself. It is a C extension that is
compiled into the Blender executable. If you open a regular Jupyter kernel —
even one that uses the exact same Python version — and you try to import bpy,
you get an import error. The module simply is not there.

So the question is: how do you get a Jupyter notebook that can actually talk to
a running Blender scene?

[SCREEN: Split view — Blender viewport on the left, JupyterLab on the right]

That is exactly what this extension solves. jupyter-blender runs an IPython
kernel directly inside the Blender process. Because the kernel lives inside
Blender, it has access to everything Blender has — including bpy. JupyterLab is
then launched as a separate subprocess and connected to that in-process kernel.
The result is a fully functional Jupyter environment where every cell you run
can read and write the live Blender scene.

[SCREEN: Type the following into a notebook cell and run it]

    import bpy
    bpy.ops.mesh.primitive_uv_sphere_add(radius=0.5, location=(1, 2, 0))

[SCREEN: The sphere appears in the Blender viewport in real time]

You type that in a cell, hit Shift+Enter, and the sphere shows up in the
viewport immediately. No copy-pasting, no reloading, no context switching. You
are writing Python in a notebook and Blender is responding live.

This opens up a whole category of workflows that were previously awkward or
impossible. You can pull in a CSV file, process it with pandas or numpy, and
turn the data into 3D geometry — all in the same notebook. You can iterate on
procedural mesh generation one cell at a time, inspect intermediate results,
and refine as you go. You get the full notebook experience — rich output,
markdown documentation, collapsible sections — while driving a live 3D scene.

That is the motivation. Now let's look at how to install it.


────────────────────────────────────────────────────────────────────────────────
SECTION 2: INSTALLATION
────────────────────────────────────────────────────────────────────────────────

[SCREEN: GitHub releases page for the project]

Installation has two parts: installing the Blender extension itself, and then
letting the extension install its Python dependencies into Blender.

Start by downloading the latest release zip from the GitHub releases page. The
file is called something like jupyter_blender-0.1.4.zip. You do not need to
unzip it — Blender will do that.

[SCREEN: Blender open, navigate to Edit → Preferences]

Open Blender and go to Edit, then Preferences. In the Preferences window, click
on the Get Extensions tab. In the top right corner of that panel, you will find
an arrow icon or a dropdown — click on it and choose "Install from Disk". A file
browser opens. Navigate to where you saved the zip file, select it, and confirm.

[SCREEN: The extension appears in the list with a checkmark]

The extension is now installed and enabled. You can close Preferences.

[SCREEN: 3D viewport, mouse moving to the right edge to open the N sidebar]

Now switch to the 3D viewport. Press N, or click the small arrow on the right
edge of the viewport, to open the sidebar. You will see a new tab called
Jupyter. Click it.

[SCREEN: The Jupyter N-panel appears with Launch and Dependencies sub-panels]

The panel has two sections. The top section is Launch — this is where you will
start and stop the notebook server. The bottom section is Dependencies — this is
where you manage the Python packages the server needs.

Right now the Dependencies section is highlighted with a warning icon, because
the required packages are not installed yet. You need to install them once
before you can use the server. Click the "Install Python Modules" button.

[SCREEN: Log box appears, pip output starts streaming]

This runs pip inside Blender's Python environment and installs three packages:
jupyterlab, ipykernel, and jupyter-client. Together they bring in everything
needed to run a notebook server — the lab interface, the kernel protocol, the
communication layer. The download is roughly 100 megabytes, so depending on
your connection this will take anywhere from a few seconds to a minute or two.
You can watch the progress in the log box that appears below.

[SCREEN: All three module names now show green checkmarks]

Once all three entries show a checkmark, the dependencies are ready.

[SCREEN: Click "Start Browser" in the Launch section]

Now click "Start Browser" in the Launch section. Blender will start the
in-process IPython kernel and launch JupyterLab as a subprocess. After a few
seconds your browser opens automatically to the JupyterLab interface.

[SCREEN: JupyterLab opens in the browser]

You are now in a full JupyterLab environment. Create a new notebook, and in the
first cell type:

    import bpy
    bpy.app.version_string

Run the cell. You will get Blender's version back as output. You are connected.

From here you can work in this notebook exactly as you would in any other
Jupyter environment — the only difference is that bpy is available and anything
you do to the scene is reflected live in the Blender viewport behind it.


────────────────────────────────────────────────────────────────────────────────
SECTION 3: HEADLESS MODE
────────────────────────────────────────────────────────────────────────────────

[SCREEN: The Jupyter N-panel, showing the "Start Headless" button]

Besides "Start Browser", you will notice a second button in the Launch section
called "Start Headless". Let me explain what that does and why you might want it.

When you click "Start Browser", the extension starts the kernel, starts
JupyterLab, and opens your default browser to the notebook interface. That is
the standalone mode — everything lives in the browser.

[SCREEN: VS Code with a .ipynb file open]

But a lot of people prefer to work in VS Code. VS Code has excellent Jupyter
notebook support built in, and it can connect to a remote Jupyter server instead
of using its own kernel. This is where headless mode comes in.

When you click "Start Headless", the extension starts the same in-process kernel
and the same JupyterLab subprocess — but it does not open a browser. Instead,
it copies the Jupyter server URL to your clipboard. That URL includes the
authentication token, so it looks something like this:

    http://127.0.0.1:10462/?token=abc123...

[SCREEN: VS Code Command Palette, searching for "Jupyter: Specify"]

In VS Code, open the Command Palette with Cmd+Shift+P on Mac or Ctrl+Shift+P on
Windows and Linux. Search for "Jupyter: Specify Jupyter Server for Connections"
and select it. A prompt appears asking for the server URL. Paste the URL that
was copied to your clipboard and press Enter.

[SCREEN: VS Code connects, a .ipynb file becomes usable]

VS Code is now connected to the kernel running inside Blender. Open any
notebook file in VS Code and you will see that the kernel indicator in the top
right shows it is connected to your Blender server. Run a cell with import bpy
and it works — VS Code is sending code to the kernel inside Blender.

[SCREEN: VS Code autocomplete working on bpy]

One advantage of this setup is that you get VS Code's full editing experience —
including autocomplete, inline documentation, and go-to-definition — while still
driving the Blender scene live.

[SCREEN: Back in the Blender N-panel, showing the "Copy Server URL" button]

If you need the URL again after you have already started the server, the panel
shows a "Copy Server URL" button whenever the server is running in headless
mode. Clicking it copies the URL to your clipboard again.

To stop the server, just click the "Stop" button. This shuts down both the
JupyterLab subprocess and the in-process kernel cleanly.


────────────────────────────────────────────────────────────────────────────────
SECTION 4: INSTALLING THE MATPLOTLIB DEPENDENCY
────────────────────────────────────────────────────────────────────────────────

[SCREEN: The Dependencies sub-panel in the Jupyter N-panel]

The Install Python Modules button takes care of the core dependencies that the
server needs to run. But in practice you will often want to use additional
packages in your notebooks — things like matplotlib for plotting, pandas for
data manipulation, or scipy for numerical work.

The Dependencies panel has a built-in way to install arbitrary packages without
leaving Blender. At the bottom of the Dependencies section you will see a text
field with a "pip install" button next to it.

[SCREEN: Click into the text field and type "matplotlib"]

Click into the text field and type the name of the package you want to install.
In this case we will type matplotlib.

[SCREEN: Click the "pip install" button]

Now click the pip install button. This runs pip inside Blender's Python
environment — the same pip that installed the core dependencies — and installs
matplotlib along with its dependencies. You will see the output streaming in the
log box.

[SCREEN: Log shows successful installation, then switch to a notebook]

Once it finishes, switch to your notebook. In a new cell, run:

    import matplotlib
    print(matplotlib.__version__)

[SCREEN: The cell runs and prints a version number]

It works. Matplotlib is now available inside the kernel running in Blender.

[SCREEN: Run a simple matplotlib plot in the notebook]

You can use it exactly as you would anywhere else. For example, you could plot
some data in a notebook cell as a regular matplotlib figure, and separately use
that same data to drive 3D geometry in Blender — all in the same notebook
session.

This pip install field accepts any package name that pip accepts — including
version pins like "numpy==1.26.0" or multiple packages separated by spaces. It
is essentially a small wrapper around pip install that targets Blender's own
site-packages, so anything you install is immediately importable in your
notebooks.

[SCREEN: Return to the full Blender + JupyterLab split view]

That is everything you need to get started with jupyter-blender: install the
extension, install the core dependencies once, start the server in browser mode
or headless mode depending on your workflow, and install any additional packages
you need through the Dependencies panel.
